Headline

The naive baseline passes 0 of 11 queries (overall accuracy 0.0). It is not a weak retriever — it is fast (p95 0.05 ms) and stays far under budget (92 of 2000 tokens injected). It fails because whole classes of requirement live outside what a similarity signal can express. A superseded fact beats the current one 80% of the time; a foreign tenant’s memory leaked into top-k on 7 of 11 queries; a health-PII record was injected 3 times, once on a query with no relevant memory at all. Every failure is structural: re-ordering the same candidate set does not add a validity field, a tenant boundary, or a PII gate. hypothesis supported

1. Hypothesis (pre-registered)

From baseline_protocol.md §1: the strongest of the simpler designs — single-signal retrieval over stored chunks — fails on the GTM workload not because its retriever is weak, but because temporal validity, resolved meaning, admission control, and tenant isolation are orthogonal to any similarity signal. If naive retrieval instead handled the adversarial conditions, Deliverable 1’s design would be over-built and should be cut. The baseline was constructed to give that outcome a fair chance (§7).

2. Baseline design

Approach D from reconstruction/failure_analysis.md, implemented literally: store every chunk → retrieve top-k by cosine similarity → inject in similarity order until a token budget is hit. Similarity is lexical TF-IDF cosine (pure stdlib, deterministic, no model download — justification and threats in the protocol). Deliberately absent, and that absence is the experiment: admission control, typed/resolved facts, temporal validity, multi-signal ranking, index-level tenant partitioning, and any abstention threshold. Fixed parameters: k=5, budget=2000, seed=20260722.

3. Workload

A fixed, seeded, adversarial-by-construction GTM dataset: 44 memories (1 PII, 3 foreign-tenant, 68% filler/duplicate), two tenants that both have an “Acme” account with near-identical notes, and 11 queries covering all six required conditions. Traps are hand-authored so a single similarity signal is forced to expose each failure — stale facts carry more query-term overlap than their corrections, so the lexical bias runs in the baseline’s favour, making this a harder test to pass, not an easier one.

4.1 Four failures, concretely (from error_examples.jsonl)

q02 — “Who is the decision maker at Acme?”  F4 + F10
Rank 1: m003 stale — “Ravi Menon, VP Engineering” (0.752). Rank 2: m102 from tenant T2 (Contoso) — “Dana Whitfield” (0.717). The correct current fact (Priya owns it now, m004) is at rank 31. A departed contact and a different customer’s decision-maker both beat the truth.
q04 — “Does Acme need SSO?”  F7 + F11
Rank 1: m005 — “I don’t think we need SSO for the pilot” (0.463). The resolved fact (“put it in — non-negotiable for production”, m006) is at rank 31. The injected context asserts the inverse of the account’s real position — and rank 3 is a health-PII record (m009: a colleague’s chemotherapy leave and personal mobile) on a question about SSO.
q09 — “Renewal date for the Globex account?”  cold-start + F10 + F11
Nothing on Globex exists; the baseline should abstain. Instead it injects m103 (T2’s confidential 85k budget), an unrelated SSO utterance, and the PII record m009. The worst possible answer to “I have nothing on file” is another tenant’s deal size plus someone’s medical data.
q01 — “What CRM does Acme use?”  F4 + F8
The entire top-3 is duplicate recitations of the stale Salesforce fact (d001/d002/d003, all 0.44). The current fact (migrated to HubSpot, m002) is at rank 14. Storing everything let one obsolete fact manufacture its own majority.

The baseline reproduced 2 of D1’s 5 unrecoverable failures (F10, F11), and both fired hard. That is the whole argument for the architecture in one line: a system can tolerate retrieving a mediocre memory; it cannot tolerate retrieving another tenant’s memory or a record it should never have stored.

What this run does NOT show — stated to keep the result honest

F1, F2 (correction across sessions; quadratic prompt growth) need a multi-turn write path; F5, F6 (summarisation loss / drift) need a summariser; F9 (deleted-yet-retrievable) needs a delete-then-requery test. A retrieval-only baseline structurally cannot exercise these — they move to the Deliverable 6 verification harness. F3 did not reproduce here: the buried EU-residency dealbreaker (m020) happened to surface at rank 1 because the query shared its terms, so q08 failed on F8/F10, not on a buried-fact miss. Reported as a non-confirmation, not glossed over. The dataset is small and author-built: it shows the existence and character of each failure, not population frequencies.

6. Proposed response — and the D1 tags it retires

Each measured failure maps to a mechanism already decided in Deliverable 2, so the baseline does more than confirm the problem — it prices the fix:

The tags that remain open (restatement-token share, summariser drift) require the mechanisms this baseline deliberately lacks and stay to measure for Deliverable 6.

7. Falsification

What would have refuted the hypothesis: high recall@k with failures concentrated in low-similarity misses — the kind a better encoder fixes. That is not what happened.

  • The gaps are validity/identity gaps, not relevance gaps. The correct fact sat at rank 31 in two separate cases while a stale or foreign chunk sat at rank 1–2. No re-ordering within a similarity score closes a 30-rank gap that encodes which fact is true or whose tenant it is — those axes are not in the score.
  • A stronger embedder makes F10 worse, not better. The cross-tenant leaks come from near-identical notes across two “Acme” accounts; higher semantic fidelity raises their similarity, so a better model leaks more confidently. Isolation cannot be earned by accuracy.
  • Cost and budget are ruled out. p95 latency 0.05 ms and 92/2000 tokens injected mean the failures are not artefacts of a starved budget or a slow index.

Honest caveats. One targeted failure (F3 buried-fact) did not reproduce on this dataset, and the abstention result is measured against a reference threshold the baseline does not apply. Neither rescues the naive design: the two unrecoverable failures it did produce (F10, F11) are sufficient on their own to reject store-everything single-signal retrieval for a multi-tenant, regulated GTM setting. The hypothesis stands, and the Deliverable 4 design must own each mechanism in §6 — each is now backed by a measured failure rate, not an assertion.

Pure Python standard library, fixed seed (20260722), no network or API key. cd experiments/naive_baseline && python3 build_dataset.py && python3 evaluate.py reproduces every failure metric here byte-for-byte; only the two wall-clock latency figures vary run-to-run (in the 3rd decimal).
