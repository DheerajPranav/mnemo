## Day 3 — Deliverable 3: Productive Failure Baseline

### Objective
Build the deliberately-naive memory baseline (Approach D), run it against a fixed adversarial GTM
workload covering all six required conditions, and **measure** its failures — replacing Deliverable 1
`to measure` tags with numbers and testing the pre-registered hypothesis that the failures are
structural (outside any similarity signal), not retrieval-tuning problems.

### Context reviewed
- Handbook §5 (process), §5.2 (required baseline shape + six workload conditions), Appendix D
  (Productive Failure Report template: Hypothesis / Baseline design / Workload / Metrics / Results /
  Failure taxonomy / Proposed response / Falsification).
- D1 `failure_analysis.md` §4 (Approach D and failure cases D-1…D-6) and §6 (the F1–F11 taxonomy,
  incl. which five are *unrecoverable*).
- D2 ADR-001/002/003 — the mechanisms each measured failure should map to.

### Work completed
- Wrote the baseline as **three pure-stdlib Python files** in `experiments/naive_baseline/`:
  `build_dataset.py` (deterministic seeded dataset, `SEED=20260722`), `baseline.py` (TF-IDF cosine
  `TfidfIndex`, budgeted injection — no filtering of any kind), `evaluate.py` (runs the baseline,
  measures every failure, writes results). Plus a `naive_baseline/README.md`.
- Authored a fixed, adversarial **44-memory / 11-query** dataset: two tenants that both have an
  "Acme" account (the F10 trap), a PII record, 68% filler/duplicate, and stale facts phrased with
  *more* query overlap than their corrections (so the lexical bias favours the wrong answer — a
  harder test to pass).
- **Ran it.** Wrote `experiments/baseline_protocol.md` (pre-registered method + threats to validity),
  and rendered `experiments/productive_failure_report.pdf` from
  `_src/productive_failure_report.html` — every number machine-produced by `evaluate.py`, none
  hand-entered.

### Observations
- **0 / 11 queries pass**, yet the baseline is fast (p95 0.05 ms) and injects only 92 of 2000 tokens.
  That combination is the finding: budget never bound and latency is negligible, so the failures
  can't be explained away as "optimise the retriever" or "give it more budget." At k=5 the problem is
  *what* gets selected and *whether it should have been stored*.
- **Emergent, stronger-than-designed result:** a foreign-tenant memory leaked into top-k on **7 of 11**
  queries — not just the 2 I built to probe F10. Cross-tenant leakage is more pervasive than the test
  intended, which is the sharpest possible argument for index-level isolation.
- The two *unrecoverable* failures the baseline can reach (F10, F11) both fired hard: PII record m009
  injected 3×, once on a cold-start query with nothing relevant on file. Retrieving another tenant's
  budget + someone's medical data as the answer to "I have nothing" is the worst-case made concrete.
- Honest non-confirmation: **F3 (buried fact) did not reproduce** — the lexical signal happened to
  surface the EU-residency dealbreaker at rank 1, so q08 failed on F8/F10 instead. Reported as such.

### Decisions and reasons
- **Lexical TF-IDF over a dense embedder.** Reason: `pip` is blocked (PEP 668) and the hypothesis is
  about what similarity *cannot express* (validity, tenant, admission), which is orthogonal to the
  signal. A stdlib baseline is byte-reproducible on any machine — worth more for a baseline than a
  marginal quality gain. Documented as a threat to validity in the protocol, with the mitigation that
  the adversarial cases are authored to favour the *stale* fact.
- **Measured pass/fail is condition-appropriate, not one global rule.** A cold-start query passes by
  abstaining; an answerable one by ranking gold above any trap with no leak and no PII. Reason: a
  single accuracy number would hide the categorically different failure types.
- **Scoped the baseline to retrieval-only and said so loudly.** F1/F2/F5/F6/F9 are deliberately out
  of scope (they need a write path, a summariser, or a deletion test) and moved to Deliverable 6.
  Reason: a productive-failure baseline earns its credibility by naming what it *doesn't* show.

### Rejected alternatives
- *A dense/model2vec embedder baseline* — rejected for v1 (blocked install + not what the hypothesis
  tests); noted as the Deliverable 4 three-arm comparison (naive / +recency / validity-filter).
- *Reporting a single overall accuracy only* — rejected; the per-failure rates are the deliverable's
  point, so the CSV carries per-query rows + an aggregate block and each failure gets its own metric.
- *Quietly dropping the F3 non-result* — rejected; reported it as a non-confirmation with the reason.

### Failures and debugging evidence
- No blockers on the run itself. The one earlier blocker (PEP 668 `pip` failures for numpy/model2vec)
  is what drove the stdlib decision; recorded in the protocol §2.1 and §8.

### Verification performed
- Confirmed all five required §5 artifacts exist: `naive_baseline/` (code + committed data),
  `baseline_protocol.md`, `baseline_results.csv`, `productive_failure_report.pdf`,
  `error_examples.jsonl`.
- Re-ran `build_dataset.py && evaluate.py` from clean — every failure/correctness metric is

  sprint_plan.md. Updated `design/README.md` (Not started → Complete).

### Observations
- The design was **more determined than expected** by D1–D3: C1–C10, the ADRs, and the D3
  failure→mechanism table left few genuinely open calls. The real D4 work was (a) committing to a
  deployment tier and storage, and (b) making isolation and deletion *structural* rather than
  procedural.
- **The small/team tier is a real design lever, not just a number.** Choosing ~2×10⁶ facts made a
  single Postgres + pgvector viable, which collapsed the F9 deletion window to a transaction and let
  isolation be a row-level invariant. Most of the architecture's safety cleanliness falls out of that
  one scope decision.
- **A new threat surfaced that D1's failure analysis could not:** memory-borne prompt injection (T4).
  Because we inject retrieved memories into the agent context, the store is an indirect-injection
  channel. Typed extraction defangs most of it, but it's the least-settled thing in v1 — logged as
  residual R4, the frontier for D6 hardening. This mirrors the D1 §5 finding that some concerns are
  undiscoverable from a purely failure-driven method.

### Decisions and reasons
- **ADR-004 — small/team envelope**, explicitly overriding D1 §4's mid-market 10⁸ assumption (which
  D1 itself flagged revisable). Reason: v1 should be implementable and measurable, not provisioned for
  peak. Recorded the scale-up triggers so it's a bounded choice, not a corner cut.
- **ADR-005 — single Postgres + pgvector**, index as a derived projection. Reason: at this tier it
  makes deletion authoritative in one transaction (biggest correctness win) and isolation enforceable
  below the API. Tier-bound, with the dual-store scale-up path recorded.
- **Conflict resolution on the write path** (ADR-001), ranker orders only valid facts. Reason: directly
  retires the D3 0.80 supersession failure; keeps the ranker simple and inspectable.
- **Isolation = RLS + constructor-scoped repository.** Reason: defence-in-depth; D3 proved the
  filter-mindset leaks (7/11) and that a better embedder makes it worse — so it must be structural.
- **Op-selector kept as *prototype*, not adopted.** Reason: its per-candidate LLM cost is the ≤15%
  budget risk; honesty demands it stay unproven until D6 measures it (threat R6).

### Rejected alternatives
- *Dual store (dedicated vector DB) for future scale* — rejected for v1; reintroduces the deletion
  window for a scale we don't have. Recorded as the explicit scale-up path instead.
- *Post-retrieval tenant filter* — rejected; it's the exact D3-measured leak pattern and degrades
  recall (foreign rows burn top-k slots).
- *A learned ranker* — rejected per D1 non-goal (no labelled data yet; would make C9 harder).
