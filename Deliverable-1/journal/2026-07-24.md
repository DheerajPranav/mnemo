## Decisions
- **Isolation as a language-level invariant, not a runtime check** — a constructor argument the caller
  can't override, mechanically enforced by a static test. RLS is defence-in-depth *behind* it (ADR-005).
- **Conflict resolution on the write path** (I3): stale facts are *removed from the candidate set*, so
  even a query with a stale fact's exact text can't surface it (verified adversarially). This is why F4
  is unreachable, not merely unlikely.
- **Gates are computed, not narrated** — each is a runnable script returning an exit code against the
  fixed D3 set, so "beat the baseline" is a measured claim.

## Productive failure + recovery (§7.3) — the M3 high-risk loop
I took a **pre-change checkpoint** (`M3-pre-highrisk.md`) before writing any read-path code, and
*predicted in writing* that the first G2 run would fail on the SSO pair (m005/m006 share the exact
`recorded_at 2025-05-09`, so a "latest-timestamp-wins" rule can't break the tie). It did — **and** it
surfaced a second, unpredicted failure (two cold-start queries didn't abstain). An **L2 DEBUG** pass
root-caused both by reading the ranker's score breakdown rather than guessing:
- the SSO tie (write-path supersession compared time only);
- stopwords + a possessive-`'s` single-char token inflated lexical similarity of irrelevant facts;
- a cross-account fact matched an account-scoped query on one shared word.
The **recovery** was five principled revisions (seq tiebreak; stopwords + typed-subject-slot ranking;
strip the account name from the lexical bag; drop single-char tokens; same-account abstain), re-running
G2 after each. G0/G1 stayed green throughout, so no rollback was needed — but the restore point existed
precisely so a regression could be reverted. Full detail: `.genesis/checkpoints/M3.md`.
