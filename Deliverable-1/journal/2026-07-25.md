# Journal — 2026-07-25 — Day 6 — Deliverable 6: Implementation and Independent Verification

Format: handbook Appendix E (context · what I did · decisions · productive failure · evidence · next).

## Context / goal
D6 asks me to build the designed system incrementally and verify it against explicit functional,
quality, operational and security requirements — with verification performed **from the specification
and acceptance criteria**, not by accepting the implementation agent's own assessment (§8.2). The
completion standard is strict: the repo can be set up and tested from its documentation, all critical
claims are backed by reproducible evidence, and **known failures and residual risks are reported
rather than hidden**.

D5 had already delivered M1–M3 (isolation, write path + PII, read path). D6 therefore had to add the
remaining §8.2 minimum scope — correction/deletion/expiry, background consolidation, decision traces —
and then verify the whole thing independently.

## What I did
1. **Extended the Genesis spine for D6**: three new store tables (`consolidation_edge`, `access_log`,
   `deletion_request`), three new invariants (I7 deletion-is-real, I8 cite-never-replace, I9
   audit-fails-closed), milestones M4/M5 with gates G3/G4, and decision **D6-DR-002**.
2. **Loop M4 — lifecycle (high-risk).** Pre-change checkpoint taken first. `lifecycle.py` (delete with
   cascade + a *measured* window, correct/restore, expiry) and `consolidation.py` (a rollup written
   under a distinct `#rollup` subject so it structurally cannot supersede its own sources). Gate G3
   green on all three sub-checks.
3. **Loop M5 — observability, 3-arm eval, red-team.** `trace.py` (per-request decision trace with
   `explain(id)` naming the exact stage that dropped a memory; audit row written *before* results, so a
   failed audit means no results) and `injection_guard.py` (deterministic detector for overt
   memory-borne instructions). Gate G4 green.
4. **Independent verification suite**: a 38-case evaluation dataset, `verify.py` that re-derives each
   §8.3 check from the spec without reading any gate verdict, `run_all.sh` to capture all evidence,
   a security report, and `final_verification.pdf`.

## Decisions
- **Postgres+pgvector+RLS ships as a reviewable artifact, not a claim** (D6-DR-002). The environment
  has no Postgres, so rather than assert an unrunnable claim I shipped `postgres_schema.sql` with the
  RLS policies and recorded RLS as **UNVERIFIED (R-P1)** with the exact command that would close it.
  An unverifiable claim in a verification report is worse than an honest gap.
- **Consolidation writes under a distinct subject.** Not a style choice: under the same subject the
  supersession rule would invalidate the very sources being summarised — F5 by construction.
- **Audit before results.** Returning first and logging after means a crash yields an unlogged read —
  exactly what an audit exists to catch.

## Productive failure — the one that matters
Gate G3 **passed on its first run**, but reported an implausible number: a 12-turn conversation thread
had only **1** source to consolidate. A green gate with a wrong-looking number is not evidence, so I
investigated instead of banking it.

The write-path supersession rule was firing on *every* `(account, subject)` collision, and
`thread`/`chatter` had no entry in the type map so they defaulted to `fact` — meaning conversation
turns superseded one another. **31 of 40 admitted memories were being silently invalidated.** That is
F5 (loss) committed by the write path, and it is the failure mode this whole project exists to prevent.

The uncomfortable consequence: **D5's read-path gate had been passing partly for the wrong reason.**
It scored 11/11 against a corpus of just 9 surviving facts — the filler noise the ranker was supposed
to prove it could handle had already been deleted by the bug. I fixed the scope (supersession applies
only to slot-valued `fact`/`preference`; events accumulate), and re-ran everything: the corpus is now
30 current facts and **G2 still passes 11/11** — the same headline, now earned. `recall_at_k` fell
1.0 → 0.778, which is the honest price of restoring the noise, and I corrected the comparison report's
wording accordingly.

A second, smaller one: I wrote the 3-arm narrative *before* running it, predicting that recency would
"move the supersession number." The measurement said otherwise — recency left supersession at 0.80 and
made **cross-tenant leakage worse (7 → 10)**, because the foreign-tenant near-duplicates are newer. I
rewrote the narrative to match the data. That is D3's prediction ("a stronger ranking signal makes F10
worse") reproduced as a measurement.
